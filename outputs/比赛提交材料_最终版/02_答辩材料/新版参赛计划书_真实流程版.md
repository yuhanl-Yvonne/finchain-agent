# 第五届中国研究生金融科技创新大赛“揭榜挂帅”赛道18参赛计划书

## 项目名称
FinChain-Agent  
基于知识图谱、图嵌入与融合模型的产业链上下游企业白名单智能筛选方案

## 一、项目摘要
本项目面向第五届中国研究生金融科技创新大赛“揭榜挂帅”赛道18“基于知识图谱与图神经网络的产业链上下游企业白名单智能筛选模型”命题，围绕低空经济产业链企业识别场景，构建了一套从数据治理、知识图谱构建、图神经网络训练到融合评分输出的完整技术链路。

与原始策划书中“拟开展”的方案不同，本版本严格按照项目已经实际完成的数据处理与模型实现流程编写。项目首先对企业基础数据、产业链数据、专利匹配数据和风险关系数据进行标准化治理；随后在 Neo4j 中构建企业知识图谱，将企业与产业链环节、区域、风险事件和专利关系进行结构化表达；在此基础上，进一步构建企业同构图，采用 GraphSAGE 学习企业图嵌入表示；最后将图嵌入与去标签泄漏后的结构化企业特征进行融合，输入 XGBoost 模型，输出白名单分数、推荐等级、推荐理由和风险提示。

项目当前已经形成可运行成果，包括：知识图谱底座、GraphSAGE 训练结果、XGBoost 融合推荐结果、Neo4j 验图查询清单、人工核验表与答辩说明材料。整体方案具备较强的可解释性、可演示性和可复现性，可服务于金融机构产业链金融场景中的企业发现、授信初筛与白名单管理。

## 二、赛题理解与项目定位

### 2.1 赛题理解
赛道18的核心目标，不是单纯构建一个企业信用评分器，而是利用知识图谱与图神经网络技术，在产业链上下游关系中识别更值得进入白名单的优质企业。其本质是一个“关系驱动”的企业筛选问题。

因此，本项目聚焦以下三个核心任务：

1. 构建能够表达产业链位置与风险关系的企业知识图谱。
2. 利用图神经网络学习企业在产业链网络中的结构表示。
3. 输出可解释、可核验、可落地的白名单推荐结果。

### 2.2 项目定位
本项目定位为“产业链白名单智能筛选与核验支持系统”。系统不替代最终人工审批，而是服务于金融机构、地方金控平台或产业主管部门在白名单初筛、重点企业发现、风险核验和推荐排序等场景中的辅助决策需求。

## 三、项目背景与现实痛点

### 3.1 项目背景
在产业链金融场景中，金融机构越来越重视围绕重点产业开展批量获客和精准授信。但传统企业风控往往更偏向财务、抵押和征信信息，对科技型中小企业、关键零部件企业、系统集成企业和新兴产业应用型企业的识别能力不足。

低空经济产业具有典型的链条长、技术密集、主体多元、场景交叉等特征，单看企业自身信息很难准确判断其产业价值和金融价值，因此更适合采用“图谱建模 + 图神经网络 + 融合评分”的方式开展识别。

### 3.2 现实痛点
结合本次项目真实处理过程，主要痛点体现在以下几个方面：

1. 企业数据来源分散，企业名称、统一社会信用代码、曾用名和标准名存在不一致问题。
2. 仅凭企业自身工商信息，很难判断其在产业链中的环节位置与协同价值。
3. 风险数据、专利数据与企业主表之间存在匹配和回填问题，需要做额外治理。
4. 仅靠传统分类模型容易忽略企业间的结构关系，难以识别处于关键链位的中小企业。
5. 白名单结果不能只给一个分数，还需要配套图谱核验路径、风险提示和人工复核机制。

## 四、建设目标
本项目的实际建设目标可概括为“五步闭环”：

1. 建立低空经济企业统一主表，形成可用于建模和图谱导入的标准企业集合。
2. 构建 Neo4j 企业知识图谱，实现企业与产业链、区域、风险、专利关系的结构化表达。
3. 基于企业同构图训练 GraphSAGE，得到可用于排序的图嵌入表示。
4. 将图嵌入与结构化画像特征融合，训练 XGBoost 白名单推荐模型。
5. 输出最终白名单分数、等级、理由和风险提示，并支持 Neo4j 图谱核验与人工复核。

## 五、数据来源与真实数据治理流程

### 5.1 本项目实际使用的数据
本项目并未停留在策划层面的“可用数据设想”，而是基于当前已经整理完成的数据文件开展处理。核心数据包括：

1. 企业主数据：`企业主表.csv`
2. 产业链整合数据：`产业链整合清单.csv`
3. 企业节点数据：`company_nodes.csv`
4. 企业-产业链关系：`rel_company_chain.csv`
5. 企业-区域关系：`rel_company_region.csv`
6. 企业-风险关系：`rel_company_risk.csv`
7. 企业-专利匹配结果：`企业-专利匹配表.csv`、`有专利企业清单.csv`
8. 企业画像数据：`样本企业画像表.csv`
9. 面板与宏观扩展特征：`39家企业画像总表.csv` 等扩展结果文件

### 5.2 真实数据治理流程
本项目的数据治理流程不是概念设计，而是已经按如下顺序落地执行：

#### （1）企业主表统一
- 以统一社会信用代码作为企业唯一主键。
- 将企业名称仅作为展示字段和辅助匹配字段，不再作为唯一键使用。
- 合并 `company_nodes.csv` 与 `产业链整合清单.csv`，补充城市、产业链位置、注册资本、参保人数、经营范围、企业标签等属性。

#### （2）企业名称标准化与实体对齐
- 对企业名称、统一社会信用代码、曾用名等字段做清洗。
- 对存在简称、空格、大小写差异、`.0` 结尾等问题的代码进行标准化。
- 在不同来源之间统一映射为“统一社会信用代码 + 企业标准名”的形式。

#### （3）风险数据回填与清洗
- 在 `rel_company_risk.csv` 中，存在部分 `company_name` 缺失但属于连续同一企业的记录。
- 对这些风险记录按照连续块进行前值回填。
- 对空 `case_id` 的风险记录进行剔除，避免导入 Neo4j 时出现 `null MERGE` 报错。

#### （4）专利与企业匹配
- 通过已有的专利匹配结果，将企业与专利信息进行关联。
- 本次项目中共形成 1226 条企业-专利关联。
- 专利匹配结果既用于图谱构建，也用于后续企业创新能力特征补充。

#### （5）特征矩阵构建
- 构建 `white_list_feature_matrix.csv` 作为结构化建模底表。
- 特征来源包括企业基础信息、地址与经营范围文本统计、专利面板统计、宏观环境变量等。
- 在特征工程阶段，主动剔除了明显标签泄漏字段，如样本标记、产业链人工结论等字段。

## 六、知识图谱构建方案

### 6.1 图谱建模对象
本项目在 Neo4j 中实际构建的图谱节点类型包括：

1. `Company`
2. `ChainPosition`
3. `Region`
4. `RiskCase`
5. `Patent`

### 6.2 图谱关系类型
实际导入的核心关系包括：

1. `(:Company)-[:BELONGS_TO_CHAIN]->(:ChainPosition)`
2. `(:Company)-[:IN_REGION]->(:Region)`
3. `(:Company)-[:HAS_RISK_EVENT]->(:RiskCase)`
4. `(:Company)-[:APPLIED_FOR]->(:Patent)`

### 6.3 导入策略与结果
- 所有导入过程均采用 `MERGE`，支持幂等追加导入。
- 企业主键统一使用 `company_id = 统一社会信用代码`。
- 导入完成后，当前图谱主要规模为：
  - `Company`: 39
  - `ChainPosition`: 3
  - `Region`: 5
  - `RiskCase`: 83
  - `Patent`: 1226

这一步为后续图嵌入训练和 Browser 验图提供了稳定底座。

## 七、图模型设计与训练流程

### 7.1 图构建方式
虽然原始策划书中曾设想更丰富的异构关系图，但结合当前数据完整度与比赛周期，本项目第一版实际采用了以 `Company` 为唯一预测节点的同构图方案。

图边构建规则为：

1. 两家公司所属城市相同，建立一条无向边。
2. 两家公司产业链环节相同，建立一条无向边。
3. 若同一对公司同时满足两种规则，则合并为一条组合边。

### 7.2 GraphSAGE 模型设置
本项目实际采用 PyTorch Geometric 实现两层 GraphSAGE，主要设置如下：

- 隐层维度：64
- Dropout：0.2
- 优化器：Adam
- 学习率：0.01
- 训练轮数上限：200
- 早停策略：patience = 20
- 训练方式：全图全批次训练

### 7.3 GraphSAGE 实际训练结果
当前 GraphSAGE 阶段结果如下：

- 节点数：466
- 边数：94532
- 正样本数：40
- 最优轮次：7
- 验证集 F1：0.8000
- 验证集 AUC：0.9947
- 测试集 F1：0.6667
- 测试集 AUC：0.9923

模型输出包括：

1. 企业级预测分数 `graphsage_score`
2. 企业级图嵌入向量 `graphsage_embedding`
3. GraphSAGE Top20 推荐名单

并已回写至 Neo4j 中的 `Company` 节点属性。

## 八、融合模型设计与真实实现

### 8.1 融合建模思路
在原始策划书中，融合模型被设计为“GraphSAGE + XGBoost / LightGBM / MLP”的第二阶段方案。结合本项目当前环境与数据条件，已经实际完成的是：

**GraphSAGE 图嵌入 + 去泄漏结构化特征 → XGBoost 融合分类模型**

### 8.2 融合特征构成
输入融合模型的特征包括两部分：

1. 结构化企业画像特征  
包括企业年龄、注册资本、参保人数、经营范围长度、宏观环境变量、专利面板统计变量等。

2. GraphSAGE 嵌入特征  
使用 GraphSAGE 输出的 64 维 embedding 作为图结构表示。

### 8.3 标签与切分方式
- 标签定义：来自 `企业主表.csv` 中的 `样本标记`
- 总样本数：466
- 正样本数：40
- 数据切分：分层切分为训练集、验证集、测试集

### 8.4 标签泄漏控制
本项目在融合建模过程中，专门排查并移除了以下可能导致不真实高分的泄漏信息：

1. `样本标记`
2. 直接标签列
3. 产业链人工判定结论类字段
4. 会直接等价于标签的辅助字段

这一步是为了保证比赛结果具有可信度，而不是仅追求表面指标。

### 8.5 XGBoost 实际结果
当前融合模型结果如下：

- 验证集 F1：0.9231
- 验证集 AUC：1.0000
- 测试集 F1：0.9231
- 测试集 AUC：1.0000

模型最终输出：

1. `xgb_fusion_score`
2. `xgb_fusion_pred_label`
3. `white_list_level`
4. 推荐理由
5. 风险提示

并已回写至 Neo4j 中的 `Company` 节点。

## 九、白名单推荐机制与可解释输出

### 9.1 白名单分层机制
项目未仅输出单一分数，而是按分数映射白名单等级：

- A 类：优先纳入白名单候选
- B 类：建议重点核验
- C 类：作为储备候选继续观察
- D 类：暂不优先推荐

### 9.2 当前推荐结果
当前主结果表为 `XGBoost融合白名单推荐表.xlsx`，其中已包含：

1. 融合白名单分数
2. 白名单等级
3. GraphSAGE 分数
4. 企业所在城市与产业链位置
5. 推荐理由
6. 风险提示

### 9.3 可解释机制的真实实现
结合目前已完成内容，项目可解释性主要通过以下方式实现：

1. 图谱路径解释：在 Neo4j Browser 中查看企业与区域、产业链、风险事件的关系路径。
2. 结果字段解释：在推荐表中直接给出推荐理由和风险提示。
3. 特征重要性解释：通过 `xgboost_fusion_feature_importance.csv` 说明融合模型主要依赖的特征。

原始策划书中提到的 SHAP 和规则引擎属于后续可继续增强的方向，但当前比赛提交版已经完成了可落地的轻量解释机制。

## 十三、风险分析与应对措施

### 13.1 数据质量风险
- 风险表现：企业名称不统一、风险记录缺失、专利匹配噪声。
- 应对措施：统一社会信用代码标准化、连续块回填、空风险记录剔除。

### 13.2 模型风险
- 风险表现：样本量较小、类别不平衡、单次切分波动。
- 应对措施：使用分层切分、正负样本权重、图模型与融合模型联合验证。

### 13.3 标签泄漏风险
- 风险表现：人工标签衍生字段可能误进入模型。
- 应对措施：在特征工程和融合训练阶段主动删除泄漏字段，并对异常完美结果进行回查修正。

### 13.4 业务解释风险
- 风险表现：模型结果虽高，但金融业务人员难理解。
- 应对措施：同步输出推荐理由、风险提示、图谱验图查询和人工核验表。
